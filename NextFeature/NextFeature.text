Google Photos and Apple Photos feel magical because they combine multiple signals (text, visual content, people, places, time) behind one search entry point, plus they guide the user with suggestions and refinements rather than asking them to build queries from scratch.

2) Core concept: Smart Presets as query templates

A “Beach” preset should not be a single text prompt. It should be a small bundle of logic:

Preset anatomy (recommended)

Preset id, name, icon

Prompt set: multiple prompts and synonyms
Example: “beach”, “sand”, “seaside”, “coast”, “ocean”, “shore”, “swimming”

Negative prompts (optional): “indoor”, “document”, “screenshot”

Signal weights:

CLIP similarity weight (visual)

Metadata weight (GPS present, month is summer, etc.)

People weight (faces present, group size, etc.)

Default threshold and sorting

Optional filters:

has_gps, has_faces, is_video, date_range, camera_model

Scope: project only (matches your earlier approach)

This is how you avoid brittle “one prompt equals one concept” behavior, because CLIP style models are powerful but not consistent across all edge cases.

3) Backend design that fits your current system (CLIP already exists)

You already log that CLIP models are available and that embeddings exist, so Smart Find should reuse that pipeline instead of adding a second AI system.

A. Query execution pipeline (deterministic and debuggable)

Resolve preset to prompt list

Embed text prompts using the same CLIP model you already use

Candidate retrieval

If your embedding store is small: brute force cosine similarity can work

If it grows: you need an ANN index (FAISS or similar) or a vector enabled DB path

Apply structured filters (SQL filters first, then re-rank)

Score fusion (weighted sum of signals)

Return a virtual album (no file moves), just a filtered view in the grid

CLIP was explicitly designed for text–image similarity and “zero shot” category matching, which is why it is the right base.

B. Store everything needed to be fast

You cannot afford to compute embeddings on the fly. The rule should be:

New assets, on ingest, get CLIP embedding once.

Smart Find only reads embeddings plus metadata.
