B) Production grade pipeline pseudocode

This is designed to be robust under rapid typing, rescans, and large libraries. It includes cancellation, query hashing, index versions, and explainability.

Data structures
struct QuerySpec:
  project_id: int
  preset_id: string|null
  text: string|null
  prompts: list<string>         # resolved from preset and or parsed from text
  facets: FacetSpec             # tri-state values
  top_k: int
  candidate_k: int              # for ANN, usually 5x to 20x top_k
  semantic_weight: float        # global default, preset can override
  metadata_weight: float
  threshold: float              # global default, preset can override
  fusion_mode: enum(max, weighted_max, soft_or)
  request_id: uuid              # for cancellation and ordering

struct FacetSpec:
  date_range: TriStateRange     # Any, Must, Must-not
  has_gps: TriStateBool
  has_faces: TriStateBool
  media_type: TriStateEnum(photo, video, screenshot, any)
  rating_min: TriStateNumber
  rating_max: TriStateNumber
  location_bbox: TriStateGeo
  people_ids: TriStateList      # optional, Phase 3 combinable with People

struct SmartFindResult:
  path: string
  final_score: float
  semantic_score: float
  metadata_score: float
  matched_prompt: string|null
  reasons: list<string>         # explainability strings
Core service
class SmartFindService:

  init(semanticService, searchService, annIndex, cache, indexVersionProvider):
    self.semantic = semanticService
    self.meta = searchService
    self.ann = annIndex
    self.cache = cache
    self.indexVersion = indexVersionProvider
    self.inflight = map<project_id, CancelToken>()

  function find(querySpec: QuerySpec) -> list<SmartFindResult>:

    # 0) Cancellation, only latest request wins per project
    if inflight contains querySpec.project_id:
      inflight[project_id].cancel()
    token = new CancelToken(querySpec.request_id)
    inflight[project_id] = token

    # 1) Normalize query
    norm = normalize_query(querySpec)
    # normalize_query resolves preset prompts, trims text,
    # sets defaults, expands synonyms if you want.

    # 2) Cache key with index version for invalidation
    idxV = indexVersion.get(project_id)
    key = hash(project_id, norm.preset_id, norm.text, norm.prompts, norm.facets,
               norm.top_k, norm.candidate_k, norm.threshold,
               norm.semantic_weight, norm.metadata_weight, norm.fusion_mode, idxV)

    cached = cache.get(key)
    if cached exists:
      return cached

    # 3) Candidate retrieval, hybrid
    # 3a) Metadata candidate pool (optional, depends on facet strictness)
    # Strategy: if user set many Must facets, use SQL to restrict.
    # Otherwise keep SQL as soft boosts only to avoid recall collapse.
    metaCandidates = []
    if should_restrict_by_sql(norm.facets):
      metaCandidates = meta.query_paths(project_id, norm.facets, limit=norm.candidate_k * 2)

    if token.isCancelled(): return []

    # 3b) Semantic candidates from ANN
    # Multi-prompt fusion: compute embedding per prompt, retrieve topN, union.
    unionCandidates = map<path, SemanticHitAggregate>()

    for prompt in norm.prompts:
      emb = semantic.get_text_embedding_cached(prompt)  # cache per prompt
      hits = ann.search(project_id, emb, topN=norm.candidate_k)
      for hit in hits:
        agg = unionCandidates.get_or_create(hit.path)
        agg.add(prompt, hit.similarity)  # keep max per prompt, also store winner

    if token.isCancelled(): return []

    # 3c) Merge candidate sets
    candidates = merge_candidates(unionCandidates.keys, metaCandidates)
    # If metaCandidates is empty, candidates is purely vector set.
    # If both exist, union them, mark provenance.

    # 4) Score and rerank
    results = []
    for path in candidates:

      # semantic score
      semAgg = unionCandidates.get(path)
      semantic_score, matched_prompt = semAgg.best_score_and_prompt()

      # metadata score, not only pass fail, can be graded
      meta_eval = meta.evaluate_path_against_facets(project_id, path, norm.facets)
      # meta_eval returns:
      # - pass_hard: bool
      # - metadata_score: float
      # - reasons: list<string>

      if violates_must_not(meta_eval): continue
      if has_hard_must_facets(norm.facets) and meta_eval.pass_hard == false: continue

      # final fusion
      final_score = fuse_scores(semantic_score, meta_eval.metadata_score,
                                norm.semantic_weight, norm.metadata_weight, norm.fusion_mode)

      if final_score < norm.threshold: continue

      reasons = []
      reasons.append("semantic: " + matched_prompt + " score=" + round(semantic_score, 3))
      reasons.extend(meta_eval.reasons)

      results.append(SmartFindResult(
        path=path,
        final_score=final_score,
        semantic_score=semantic_score,
        metadata_score=meta_eval.metadata_score,
        matched_prompt=matched_prompt,
        reasons=reasons
      ))

    # 5) Sort, cut top_k
    results.sort_by(final_score desc)
    results = results.take(norm.top_k)

    if token.isCancelled(): return []

    # 6) Cache and return
    cache.set(key, results, ttl_seconds=120)  # TTL is fine if versioned by idxV
    return results
Notes you should implement immediately

ANN index for embeddings, otherwise CLIP similarity becomes slow as libraries grow. FAISS is a standard choice for similarity search and ANN indexing.

Cache must include index version, not only TTL, to prevent stale results after rescans.

Cancellation ensures old searches do not overwrite new results, essential under debounce.

C) Scoring fusion strategy used by large photo systems

Large scale retrieval systems almost always use a two stage or three stage approach:

candidate retrieval (fast, high recall)

reranking (slower, higher precision)

optional business logic boosts and filters

For your photo app, the equivalent is:

metadata retrieval and filtering (SQL)

vector retrieval (ANN, CLIP)

fusion and rerank in SmartFindService

This is a hybrid search pattern, combining lexical or structured filters with vector similarity.

1) Candidate retrieval, union, then rerank

Recommended:

Retrieve topN from ANN for each prompt

Union candidates across prompts

Optionally union with SQL candidates when facets are strict

Then rerank all candidates with fused scoring

This gives you synonym resilience, your multi prompt max fusion is already heading there.

2) Multi prompt fusion strategies

You mentioned max score fusion. Keep it, but offer two additional modes for power users, even if hidden in settings.

Let s_i(p) be the similarity score for candidate image i under prompt p.

Mode A, Max, simple and robust:

S_sem(i) = max_p s_i(p)

matched_prompt is argmax prompt

Mode B, Weighted max, reduces prompt noise:

S_sem(i) = max_p ( w_p * s_i(p) )

where w_p can be 1.0 for the main prompt, 0.7 for synonyms, 0.5 for broad prompts

Mode C, Soft OR, rewards multiple prompt agreement:

S_sem(i) = 1 - Π_p (1 - clamp01(s_i(p)))

This increases score if the image matches multiple prompts moderately well.

3) Metadata scoring should be graded, not only pass fail

Instead of only hard SQL filtering, treat metadata facets as a score contribution, except when user explicitly sets Must.

Example:

has_gps true adds +0.05

rating >= 4 adds +0.10

date in last year adds +0.03

media_type mismatch subtracts, or hard excludes if Must

This mirrors the “predictable” behavior users expect from mainstream apps that combine multiple hints, rather than acting like a strict database query. iPhone Photos and Google Photos explicitly support searching by dates, places, categories, and combinations, which implies the system tolerates partial matches and multiple signals.

4) Final fusion formula

A production friendly fusion that is easy to tune and explain:

Normalize semantic similarity to 0..1:

S = clamp01((cos_sim + 1) / 2) or your existing normalization

Normalize metadata score to 0..1:

M = clamp01(meta_score)

Fuse:

Final = α * S + (1 - α) * M + Boosts - Penalties

Where:

α defaults to 0.8 for concept searches (Beach, Wedding)

α defaults to 0.4 for utility searches (Screenshots, Videos) because metadata is more reliable

Boosts and Penalties are small, bounded, and logged in reasons

5) Calibration and thresholds

Do not hardcode one threshold for all presets.

Best practice:

global default threshold

per preset override

dynamic backoff if results are empty:

if result count is 0, lower threshold slightly and show “expanded results” chip, user can undo

This keeps the system from returning nothing, which users interpret as failure.
