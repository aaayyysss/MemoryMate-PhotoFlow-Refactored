Sidebar Evolution: Assessment & Recommendation
Your proposed "One entry point, three layers" architecture is correct in direction but needs sequencing discipline to avoid opening construction sites that block each other. Here's my assessment grounded in your actual codebase:

What you have now (8 sections)
Folders | Dates | People(+Groups) | Videos | Duplicates | Devices | Locations | Quick

All sections follow BaseSection with load_section() + create_content_widget() + lazy threading. This is a solid, tested pattern.

Your proposal mapped to reality
Your Layer	Maps to	Ready?	Effort
Layer 1 (Instant Search)	New SearchSection extending BaseSection	No. CLIP embeddings + semantic_search_widget.py exist but aren't wired to accordion	Medium
Layer 2 (Explore/Browse)	Your existing 8 sections already ARE this layer	Yes, done. People, Dates, Folders, Locations = the explore layer	None
Layer 3 (Builder/Saved)	New SavedSearchSection + saved_searches table + QueryContract	No. No DB schema, no service, no UI	Large
My recommendation: Proceed, but in this order
Phase 1 — Query Contract first (the foundation)
Define a SearchQuery dataclass before building any UI:

@dataclass
class SearchQuery:
    project_id: int
    text: str = ""                    # semantic/keyword search
    filters: dict = field(default_factory=dict)  # media_type, date_range, etc.
    mode: str = "semantic"            # "semantic" | "keyword" | "metadata"

This is cheap (one file, zero UI risk) and becomes the contract everything else speaks.

Phase 2 — SearchSection (Layer 1)

Extends BaseSection, goes at position 0 in the accordion
Text input + chip bar (People, Places, Recent, etc.)
Chip clicks emit SearchQuery objects via signal
GoogleLayout gets apply_query(query) that routes to existing methods (_load_photos with filters, _show_filtered_view, etc.)
This is immediately useful because you already have CLIP semantic search
Phase 3 — DB schema + SavedSearchSection (Layer 3)

saved_searches(id, project_id, name, query_json, pinned, created_at, updated_at)
recent_searches(id, project_id, query_text, last_used_at, use_count)

New SavedSearchSection extending BaseSection
Stores queries as JSON, displayed as clickable sidebar items
Context menu: rename, delete, duplicate, pin
Phase 4 — Label Extraction for instant chips (your Phase 3)

Background LabelExtractionJob using CLIP → top-K labels per photo
asset_labels(asset_id, label, score, model, updated_at) table
Chips ("Beach", "Wedding", "Mountain") become SQL filters, not CLIP inference
This is what separates "demo" from "product" — you're exactly right
What NOT to do
Don't build the Rule Tree Builder (Phase 4/Layer 3) yet. You'll over-engineer it before you know what queries users actually save. Start with free-text + filters, add rules later.
Don't consolidate GroupService + PeopleGroupService into a service registry yet. It works, the patterns are clear. Service injection is a refactor you do when you add project switching, not now.
Don't change BaseSection to add search awareness. SearchSection should be a peer section with its own signal contract, not a modification to the base class.
The single most important insight
Your existing 8 sections = Layer 2 (Explore). You don't need to rebuild them. SearchSection sits on top as a new section, SavedSearchSection sits below as another new section. Both extend BaseSection. The accordion pattern already supports N sections — you just add 2 more.
