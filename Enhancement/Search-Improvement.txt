Here is the closest practical reconstruction of the Apple Photos search pipeline, based on Apple’s public product behavior, Apple Support / Developer documentation, and Apple patents around digital asset search and knowledge-graph based categorization. It is not Apple’s internal source code, but it is close enough to use as a design blueprint.

1. The real top-level architecture

Apple’s likely search stack is not “one CLIP search per preset.” It is a layered pipeline:

Asset ingestion
→ metadata extraction
→ on-device vision analysis
→ entity/tag graph construction
→ search query parsing
→ candidate retrieval
→ rule/gate filtering
→ ranking
→ UI presentation with suggestions and refinements.

The strongest clue is Apple’s own public behavior: Photos search exposes people, pets, places, dates, categories, and groups, while Apple patents explicitly describe a knowledge graph of assets and categories, with UI elements generated from that graph.

2. Stage by stage, what Apple is probably doing
Stage A, ingestion and canonical metadata

Every photo/video first gets normalized into a canonical asset record:

file identity

timestamp

GPS / place

media type

album / collection membership

favorite / hidden / deleted state

technical dimensions and orientation.

This part is obvious from Photos product behavior and Apple Support search features around moments, people, places, and categories.

Stage B, on-device vision analysis

Apple’s Vision stack publicly exposes detectors for faces, rectangles, text, barcodes, and objects, and Apple positions much of Photos intelligence as on-device. In practice, Apple Photos likely runs multiple analyzers per asset:

face detection and clustering

person / pet grouping

scene classification

object classification

text detection / OCR

screenshot-like / document-like cues

saliency / key-photo quality signals.

Stage C, semantic category graph

This is the piece many developers miss. Apple’s patents around digital asset search describe building a knowledge graph / metadata network where assets connect to keyword tags, categories, and related concepts. That means Apple likely does not rely on one raw embedding alone. It builds a graph such as:

photo
↔ person: Ammar
↔ place: Munich
↔ category: beach
↔ category: sunset
↔ text: invoice
↔ event/group: family dinner.

That graph is exactly why Apple can show suggestions like people, places, categories, groups, and likely support mixed search behavior without the user understanding the underlying model.

Stage D, query understanding

When the user types something, Apple likely decomposes the query into multiple possible intents:

person intent

place intent

date/time intent

category intent

OCR/text intent

event/group intent

free semantic intent.

For example, a query like “Ammar at the beach in 2024” is probably decomposed into:

person = Ammar

category = beach

year = 2024

optional place prior from GPS / map clusters

semantic expansion around shore, seaside, coast.

That is much closer to Apple’s likely behavior than a single cosine similarity search. This is partly inferred from Apple’s public feature set and partly supported by the knowledge-graph patent language.

Stage E, candidate retrieval, not one path but several

Apple almost certainly retrieves candidates from several indexes in parallel:

metadata index

people/pets index

place index

OCR/text index

category / scene / object index

group / co-occurrence index

semantic similarity index.

Then it unions or intersects them depending on query intent. This is the key architectural difference between a good photo app and a truly strong one. Apple’s UI and patent material strongly imply multi-source retrieval rather than one monolithic search pass.

Stage F, hard gates

After candidate retrieval, Apple-style systems apply structural gates where category purity matters:

Screenshots: require screenshot signal

Documents: require document-like signal, reject face-heavy photos

People events: require face presence

Map/location results: require reliable GPS or place inference

Videos: require media_type=video.

This is the part you are now discovering in your own app. Apple’s product behavior strongly suggests these gates exist, because categories like screenshots, people, groups, and places behave much more structurally than pure semantic search would allow.

Stage G, ranking

This is where Apple probably combines many signals, something like:

final rank =
semantic relevance

category confidence

person match confidence

OCR match confidence

location match confidence

recency prior

user affinity, favorites, hidden state

“best representative” / key-photo quality.

Apple’s search patents mention ranking and presentation of search-related properties, and the Photos UI clearly favors representative thumbnails and suggestion chips instead of just dumping raw matches.

Stage H, presentation layer

Apple then surfaces:

suggestions first

grouped results

representative thumbnails

people/pets collections

group photos

place browsing

category browsing.

That means the visible search UX is really the last 10 percent of a much deeper indexing system.

4. What Apple likely does that your current app still does not

Three important things:

First, category confidence is stored separately from semantic similarity

Apple almost certainly does not treat “document”, “screenshot”, “wedding”, and “mountain” as the same kind of signal. Category families are different. That matches what you are now moving toward with preset families.

Second, graph relationships matter

A group photo, a named person, a place cluster, and OCR text are not just flat tags. They are connected entities in a graph. Apple’s knowledge-graph patent language points directly in that direction.

Third, UI suggestions are generated from the indexed graph, not hardcoded only

Apple patents explicitly describe UI elements for search categories generated from the knowledge graph of the asset collection. In plain language, the library itself helps decide which suggestions are worth showing.

5. What this means for your next engineering step

You should not build fully separate pipelines per preset. Your earlier instinct there would create duplication, drift, and maintenance pain.

You should build:

one orchestrator

one query planner

one shared multi-source retrieval layer

one family-aware gating layer

one ranking layer

Then each preset only changes:

semantic prompts

preset family

hard gates

ranking profile

That is the Apple-like design.

8. My recommendation for your project, in priority order

First, finish the family-aware gating and ranking profiles.

Second, add OCR/text indexing as a first-class search source for Documents and Screenshots.

Third, add entity graph tables for people, places, groups, and categories.

Fourth, move toward free-text query planning, not only preset matching.

Fifth, only then optimize prompts further.

Prompt expansion helps recall, yes, but it is not the deepest win. The deeper win is the graph + family-aware search planner. That is much closer to Apple Photos.

class GateEngine:
    def apply(self, candidates, plan, meta):
        kept = []
        dropped = {
            "require_screenshot": 0,
            "exclude_screenshot": 0,
            "require_faces": 0,
            "exclude_faces": 0,
            "require_gps": 0,
            "require_pick": 0,
            "min_edge_size": 0,
        }

        gates = plan.hard_gates or {}

        for c in candidates:
            m = meta.get(self._norm(c.path))
            if not m:
                continue

            if gates.get("require_screenshot") and not bool(m.is_screenshot):
                dropped["require_screenshot"] += 1
                continue

            if gates.get("exclude_screenshot") and bool(m.is_screenshot):
                dropped["exclude_screenshot"] += 1
                continue

            if gates.get("require_faces") and (m.face_count or 0) <= 0:
                dropped["require_faces"] += 1
                continue

            if gates.get("exclude_faces") and (m.face_count or 0) > 0:
                dropped["exclude_faces"] += 1
                continue

            if gates.get("require_gps") and not bool(m.has_gps):
                dropped["require_gps"] += 1
                continue

            if gates.get("require_pick") and (m.flag or "").lower() != "pick":
                dropped["require_pick"] += 1
                continue

            min_edge = gates.get("min_edge_size")
            if min_edge:
                w = m.width or 0
                h = m.height or 0
                if min(w, h) < min_edge:
                    dropped["min_edge_size"] += 1
                    continue

            kept.append(c)

        return kept, dropped

    @staticmethod
    def _norm(path: str) -> str:
        import os
        return os.path.normpath(path).lower()

Proposed skeleton
def search(self, plan: QueryPlan, top_k: int) -> OrchestratorResult:
    meta = self.search_graph.get_asset_meta(self.project_id)

    candidates = self.retriever.retrieve(plan, top_k=top_k * 4)

    gated, dropped = self.gate_engine.apply(candidates, plan, meta)

    scored = self.ranker.score_many(gated, meta, plan)

    scored = self.deduplicator.deduplicate(scored, meta)[0]

    scored = self._revalidate_after_dedup(scored, plan, meta)

    scored.sort(key=lambda r: r.final_score, reverse=True)
    scored = scored[:top_k]

    facets = self.facet_service.compute([r.path for r in scored], meta, plan)

    return OrchestratorResult(
        paths=[r.path for r in scored],
        total_matches=len(scored),
        scored_results=scored,
        facets=facets,
        query_plan=plan,
    )
