What your logs show is not a tuning problem, it is a taxonomy problem: ViT-B/32 similarity scores overlap across “documents” and “normal photos”, so you need structural discriminators (hard filters) plus a small second stage re-rank (soft signals). That is exactly how the big apps behave: they combine semantic search with metadata and content signals (people, places, objects) rather than trusting one score.

Below is a concrete, step-by-step plan, tied to the files you already have (based on the snippets you pasted and the code in services/search_orchestrator.py that already contains _get_project_meta() and _detect_screenshot()).

Goal: every preset becomes {retrieve → hard gates → soft boosts → final slice}. This is the iPhone, Google Photos, Lightroom mental model: a category is not just a query string, it is a pipeline.

Step 3: Fix “Documents” with a proper 3-part gate, not a threshold

Documents: recommended pipeline

Negative prompts: ["selfie","portrait","person","face","group photo","landscape","mountain","beach","city","pet"]

Hard gates (structural)

exclude_faces: True
Use your existing face coverage from face_crops (you already fixed the table name, good).

exclude_screenshots: True
Use your existing _detect_screenshot() and meta["is_screenshot"].

require_document_visuals: True (new)

This one is the missing piece that will eliminate “random photos with some texture” being accepted as documents.

Soft boosts (ranking)

A4-ish ratios (portrait ~ 0.70 to 0.78), receipts (very tall), screenshots usually 9:16 (but you already excluded them)

boost if EXIF indicates scan apps (later enhancement)
